# Project Strand — Security & Finance Update (2026-07-03)

This update closes the cross-tenant security holes found in the audit and wires the
previously-deferred finance links into the general ledger. It is **build-verified**
(`npx tsc --noEmit` and `npm run build` both pass). No database migration is required —
the schema is unchanged; the app applies it on boot as before.

## How to apply
1. Unzip over your local working copy (it overwrites source files only; it does not
   touch `.git`, `node_modules`, `.env`, `.pgdata`, or `.uploads`).
2. Review the diff, then commit and push:
   ```
   git add -A && git commit -m "fix: close cross-tenant IDOR; wire finance links to the ledger"
   git push
   ```
3. Render redeploys on push. **Before/after deploy, set these env vars** in the Render
   dashboard: `AUTH_SECRET` (must be a long random value — confirm it exists), and
   `CRON_SECRET` (any long random string) to lock the renewals cron endpoint.

## 1. Security fixes

### Cross-tenant IDOR (the critical issue)
Added ownership-assertion helpers at the top of `src/app/actions.ts` and called them in
every affected server action, so a client-supplied record id is always tied back to the
caller's authorized project/org. Also scoped the previously-unscoped `DELETE`s via
subqueries. Covered actions: requisitions (submit/decide/disburse/record-expenditure/
attachment/voucher/approve-voucher), budget (add/update/delete line, add expenditure,
setup sections, applySuggestions), finance-ops (issue/void invoice, depreciate asset,
create/edit receipt), HR (employee education/policy add+delete, comp, employee &
collaborator logins), collaborator project link/unlink, and logframe/SOW/report CRUD.
The effect is that neither an existing tenant nor a new self-signup can reach another
organisation's records.

### Project-reference injection (secondary hardening)
An adversarial re-review found a lower-severity class where a client-supplied
`projectId`/`outputId` could be stamped onto the caller's *own* new/edited row without
checking the project belonged to their org (data-integrity, plus a currency leak in
payment slips). Added a `projectInOrgOrNull(projectId, orgId)` resolver and applied it to
invoice create/edit, fixed-asset create, payment-slip create, receipt create/edit,
timesheets (manager + self-service), purchase requests (manager + self-service), and
employee-compensation; the manager timesheet also now checks the employee is in-org, and
activity→output linking validates the output belongs to the project.

### Authentication hardening
- **Login rate-limiting** — `src/lib/rate-limit.ts` (new) + `signIn` in `actions.ts`.
  After 5 failed attempts within 15 minutes for a given email *or* source IP, that key is
  locked for 15 minutes. In-memory (fits Render's single free-tier instance); move it to
  the DB/Redis if you scale to multiple instances.
- **Server-side session expiry** — `src/server/auth.ts` now validates the signed
  timestamp in the session token and rejects tokens older than 7 days, and compares the
  signature in constant time.
- The login page now shows a distinct "too many attempts" message.

## 2. Finance links now posting to the general ledger

Each posting is a **no-op until the org has run "Initialise ledger"** (chart of accounts),
de-duplicates by source document so retries never double-post, and converts to the org's
base currency. New reusable posters live in `src/server/services/ledger.ts`
(`postCashPayment`, `postCashReceipt`, `postFundsTransfer`, `postPayrollToLedger`,
`postVendorBillToLedger`, `postVendorBillPayment`).

| Event | Journal entry (accounts by standard code) |
|---|---|
| **Payroll finalised** (`hr.ts`) | DR 5000 Salaries (gross); CR 2200 Payroll liabilities (deductions); CR 1000 Cash (net) |
| **Vendor bill raised** (`procurement.ts`) | DR expense (bill's account, else 5200); CR 2000 Accounts payable |
| **Vendor bill paid** (`postVendorBillPayment`, ready to wire to a pay-bill action) | DR 2000 AP; CR 1000 Cash |
| **Sub-award disbursement** (`addSubawardPaymentAction`) | DR 5900 Indirect; CR 1000 Cash — tagged to the funding project |
| **Per-diem paid** (`markPerdiemPaidAction`) | DR 5100 Travel; CR 1000 Cash — tagged to the claim's project |
| **Grant/funding receipt** (`recordReceiptAction`) | DR 1000 Cash; CR 4000 Grant income — tagged to the agreement's project |
| **Petty-cash replenishment** (`replenishPettyCashAction`) | DR 1010 Petty cash; CR 1000 Cash |
| **Investment placement** (`createInvestmentAction`) | DR 1300 Short-term investments; CR 1000 Cash |
| **Investment interest** (`recordInvestmentMovementAction`) | DR 1000 Cash; CR 4900 Other income |
| **Investment withdrawal/maturity** | DR 1000 Cash; CR 1300 Short-term investments |

Added account **1300 Short-term investments** to the standard chart. Also: **every new
project now gets a budget scaffold at creation** (standard donor sections), so the
budget → requisition → expenditure → ledger chain exists from day one; the lazy
budget-creation paths were made get-or-create so no project ends up with two budgets.

### Assumptions worth reviewing
- Payroll treats **all deductions as one payroll-liability line** (2200). If you track
  PAYE and NSSF (and staff savings) separately, split that credit across distinct accounts.
- Sub-award disbursements post to **5900 Indirect**; change the expense code if you'd
  rather they hit a dedicated sub-grant expense account.

## 3. Deliberately NOT auto-implemented — these need your accounting policy

I did not guess at these three, because a wrong default would put bad numbers in your
books. Each is a small, well-scoped follow-up once you decide the policy:

1. **Reserve-fund movements → GL.** A reserve allocation is an appropriation of fund
   balance (equity), not necessarily a cash movement. The correct entry (e.g. DR 3000
   Unrestricted / CR 3100 Restricted) depends on how you want reserves represented.
   Currently reserves are tracked in `reserve_movement` but not posted.
2. **GRN → Inventory valuation → GL.** Receiving goods should raise stock and a stock
   asset, but that needs a mapping from purchase-order lines to inventory items and an
   inventory asset account. `createGRN` records receipt quantities; it does not yet value
   stock or post to the ledger.
3. **Payroll → per-project salary allocation.** Charging each employee's cost to specific
   project budget lines needs a cost-driver rule (which line, split by `effort_pct` or
   timesheet hours). The org-level payroll GL entry above is posted; per-project
   allocation is left for your policy. `employee_project` and `timesheet.project_id`
   already hold the data needed.

Tell me the policy for any of these and I'll wire them the same way.
